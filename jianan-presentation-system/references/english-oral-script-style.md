# English Executive Oral Script Style

Use this reference when Jianan asks for a full-English oral script, an English script polish, or a Chinese-to-English presentation adaptation. Pair it with `oral-script-guide.md`; this file defines the English delivery style, compression rules, and memorable landing lines.

---

## 1. Target Voice

The target voice is a hybrid:

- **A: Executive clarity** - clean logic, short sentences, low jargon.
- **B: Product demo energy** - concrete workflow language, visible before/after contrast, user proof.
- **C: Strategic closing** - use sparingly on the final 1-2 slides, where the script explains why the capability matters beyond the first project.

The English version should sound polished but spoken. It should feel like a strong live briefing, not a translated memo.

---

## 2. Core Rhythm

For each slide, rebuild the message in this order:

1. **What changed** - the old work pattern vs. the new AI-assisted pattern.
2. **Why it matters** - time saved, quality stabilized, scale unlocked, or trust built.
3. **Proof** - numbers, user feedback, demo result, or workflow validation.
4. **Landing line** - one short sentence that sticks. Use only when the slide needs emphasis.

Default sentence shape:

```text
Before, [manual workflow].
Now, [AI-assisted workflow].
The value is [specific impact].
And the key point is: [landing line].
```

Keep paragraphs short. As a rule of thumb, no paragraph should exceed four short spoken sentences.

---

## 3. Narrative Judgment Rule

Do not force every slide into the same template. The goal is not "capability spectrum" specifically; the goal is a script that has **logic, point of view, speaking rhythm, and a memorable conclusion**.

For each slide or section, first decide what kind of thinking the audience needs to feel:

- **See the structure** - use a spectrum, matrix, 2x2, maturity ladder, or layered model.
- **Feel the contrast** - use before/after, manual vs. AI, old workflow vs. new workflow.
- **Trust the proof** - use real-user feedback, validation numbers, edge cases, or expert review.
- **Understand the boundary** - state what AI can do, what it should not do, and what must stay human-led.
- **Move to action** - convert the result or boundary into the next step, resource ask, or strategic decision.

High-quality oral script should feel **conclusion-led, not description-led**. A strong paragraph usually contains:

```text
[Sharp frame] -> [Specific proof] -> [Why it matters] -> [Landing sentence]
```

Use named patterns only when they genuinely fit the case. If the slide is about multiple modules, a capability spectrum may work. If the slide is about one painful workflow, before/after may be stronger. If the slide is about risk, boundary-and-control may be stronger.

---

## 4. What To Avoid

Avoid dense translated phrases that sound written instead of spoken:

- "From the perspective of..."
- "In the broader context of..."
- "It has multi-layered significance..."
- "This provides an important foundation for..."
- "We can further empower the transformation of..."
- Long noun stacks like "cross-functional workflow capability construction"

Prefer direct spoken phrasing:

| Dense | Better |
|------|--------|
| From the perspective of shared demand across BUs | Across BUs, this is not a one-off need |
| In the broader Q context | For Q, this can be an early proof point |
| The tool reduces man-days by half | The work dropped from 8 man-days to 4 |
| More importantly, it helps us build... | More importantly, it shows we can build... |

## 5. Landing Line Bank

Use one landing line every 3-4 slides. Do not overload the script with slogans.

### Expert + AI

- The tool drafts. Experts decide.
- We are not replacing expertise. We are giving expertise more room to work.
- AI should not take experts out of the loop. It should take repetitive work off their plate.
- Experts teach once. The machine can execute many times.

### Automation Logic

- If a piece of work can be verified, there is room to automate it.
- Anything repeatable can be structured. Anything structured can be improved.
- The value is not one report saved. The value is one workflow proven.
- Every gap becomes feedback. Every feedback makes the next run stronger.

### Product + Strategy

- CER is not the end point. It is the first proof point.
- One validated workflow can become a reusable capability.
- We are not just building a tool. We are learning how the organization can build AI capability from real work.
- The first use case saves time. The repeatable model builds advantage.

### Trust + Human-in-the-loop

- Trust is not claimed. It is designed into the workflow.
- The AI moves fast, but the expert keeps control.
- Automation does not remove review. It makes review more focused.

## 6. English Adaptation Workflow

When converting a Chinese oral script to English:

1. **Preserve the slide structure** - keep the same slide numbers, timing, and core proof points.
2. **Compress the Chinese logic** - find the spoken claim behind each paragraph before translating.
3. **Rebuild in English** - write as if presenting live to executives, not as a literal translation.
4. **Add sparse landing lines** - one every 3-4 slides, only at natural turning points.
5. **Humanize** - remove inflated AI-writing patterns, long transitions, repeated triads, and abstract promotional language.
6. **Alignment check** - verify that numbers, names, user quotes, resource asks, and technical terms match the source.

## 7. Slide-Level Guidance

### Opening / Progress

Lead with the real status:

- The tool has moved beyond prototype.
- Internal users are already using it.
- One CR report generated with the tool was submitted for project initiation in mid-April.
- The work dropped from 8 man-days to 4.

Possible landing line:

> This is no longer only a demo. It is already touching real registration work.

### Problem Scale

Make the scale concrete and simple:

- Around 25 CER reports per year.
- CT accounts for around 12.
- From January to April this year, 6 products were already running CER work.
- Each BU needs roughly 1 man-month to migrate the first workflow, and the cost should go down as reuse increases.

Possible landing line:

> Across BUs, this is not a one-off need. It is a repeatable workload.

### Wow Moments

Use before/after contrast, then proof:

- Keyword replacement: from 2-4 hours to seconds.
- Comparison table: from more than 2 days to 3 minutes.
- Difference analysis: from expert-heavy drafting to draft-first review.
- Literature review: from days to minutes.

Possible landing line:

> The tool does not try to make final decisions alone. It creates a better starting point for expert review.

For modular AI products, consider the **capability spectrum** pattern when it fits the case and the slide needs to feel executive-level instead of feature-list-like:

1. Name the workflow tension in a sharp frame, such as "triple-high": high compliance, high repetition, high expert dependency.
2. Break the system into task types, each mapped to a different AI capability profile.
3. Move from the most automated module to the deepest collaboration module.
4. Attach one proof point to each module: time, accuracy, validation count, or compliance baseline.
5. Close with a division of labor between AI and experts.

Reusable shape:

```text
[Workflow] is a "[tension name]" workflow - [pressure 1], [pressure 2], [pressure 3].
Our approach is to break it into [N] task types, each matched to a different AI capability.
One - [module]. The most [capability profile]. [proof].
Two - [module]. The most [capability profile]. [proof].
Three - [module]. The deepest [collaboration/intelligence profile]. [proof].
Four - [module]. The highest [throughput/reuse profile]. [proof].
[N] task types, [N] capability profiles - AI-led production, expert-led judgment.
```

Good landing lines:

- Four task types, four capability profiles - AI-led production, expert-led judgment.
- Experts should spend their time on judgment, not on reading.
- For compliance, this is the line that matters most.

### User Voice

Quote users directly where possible. Do not over-explain the quote. Let it carry the trust.

Useful bridge:

```text
This feedback matters because it came from the workflow itself.
The users were not judging a concept. They were testing whether the output could fit their real work.
```

Possible landing line:

> Trust is not claimed. It is designed into the workflow.

For evidence pages after a capability-spectrum slide, use the **two-end validation** pattern:

1. Pick feedback from the two extremes of the spectrum, not every feature.
2. Use the most automated end to prove immediate workflow value.
3. Use the most AI-intelligent end to prove collaboration value and state the boundary.
4. Convert the boundary into the next-step logic.

Reusable shape:

```text
I picked feedback on the two ends of the spectrum - [most automated module] and [deepest AI module].
First, [module] - the most automated end. [User feedback + value felt immediately].
Second, [module] - the deepest AI end. [Usable scope].
But the boundary is clear: if [missing data] is not in the system, AI should not invent it.
The takeaway: once we can bring complete information into the system and verify it, there is still room to automate further.
```

### Strategic Close

Shift from "what the tool does" to "what this mode proves":

- Real user need.
- Validated workflow.
- T1 core team.
- More complete backend support.
- Roughly 3 man-months for the next PTR phase, depending on final scope.

Possible landing line:

> The first use case saves time. The repeatable model builds advantage.

## 8. QA Checklist

Before delivering an English script:

- Same slide structure and timing as the source.
- All numbers match the latest approved version.
- No Chinese residue in the English-only file.
- No paragraph longer than four short spoken sentences unless intentionally used for a quote.
- One memorable landing line every 3-4 slides, not every slide.
- Strategic tone appears mainly in the closing, not throughout the whole script.
- Technical terms stay precise: CER, PTR, CR, BU, CT, human-in-the-loop, false negative, automation coverage.
- User quotes are clearly marked as quotes or paraphrases.
